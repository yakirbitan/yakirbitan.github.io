What IS "The Cloud"? 🤔
It's literally someone else's computer. But a LOT of them — managed for you.
🏠
Before Cloud (On-Premise)

You buy servers, put them in a room, plug in cables, install OS, update software, replace broken disks, pay for electricity 24/7, hire someone to guard them.

💰 $500K-$2M upfront + ongoing costs
☁️
With Cloud

Click a button → get a server in 30 seconds. Need 100 more? Click again. Done with them? Delete. Pay only for what you use, by the hour or second.

💰 $0 upfront, ~$0.02/hour per server

💡 The real magic: Not just renting servers — it's 200+ managed services: databases, AI, queues, DNS, security, monitoring... all available via API in seconds. You build, they operate.

The Big 3 Cloud Providers ☁️
AWS, GCP, and Azure dominate ~65% of the market

🎯 Which to learn? Start with AWS (most jobs require it). Add GCP if you do data/ML. Add Azure if your company uses Microsoft. The concepts are 90% identical — only service names differ.

Region

A geographic area (e.g., "us-east-1" = Virginia). Each region has 3+ Availability Zones. Choose a region close to your users for low latency.

Availability Zone (AZ)

A separate data center within a region, with its own power, cooling, and network. If one AZ burns down, others keep running. Always deploy across 2+ AZs!

🔥 Real example: In 2017, an AWS engineer accidentally took down S3 in us-east-1. Half the internet broke. Lesson: multi-region = survival.

🐳 What IS a Container?

A lightweight, isolated package containing your app + all its dependencies. Like a VM but shares the host OS kernel → starts in milliseconds instead of minutes. Docker creates containers, Kubernetes orchestrates thousands of them.

⚡ Serverless explained:

Upload a function → it runs when triggered (HTTP request, file upload, timer). 0 requests = $0 cost. 1 million requests ≈ $0.20. You don't think about servers AT ALL. The cloud handles scaling, patching, everything.

🏰 VPC (Virtual Private Cloud)

Your own private network in the cloud. You define IP ranges, subnets (public/private), firewalls. Nothing gets in or out unless YOU allow it.

⚖️ Load Balancer

Distributes traffic across multiple servers. If one server dies, LB sends traffic to healthy ones. Essential for high availability.

🌐 DNS (Route 53 / Cloud DNS)

Translates domain names (myapp.com) → IP addresses. Can do geographic routing: users in Europe → EU servers, US users → US servers.

🚀 CDN (CloudFront / Cloud CDN)

Caches your content at 400+ edge locations worldwide. User in Tokyo gets your image from a Tokyo cache instead of a Virginia server. ~10x faster.

📜 IAM Policy (JSON) — Example:
{
  "Effect": "Allow",
  "Action": ["s3:GetObject", "s3:PutObject"],
  "Resource": "arn:aws:s3:::my-bucket/*"
}

Translation: "Allow this identity to read and write files in my-bucket." That's it. No access to delete, no access to other buckets.

🔑 Golden Rule: Least Privilege

Give the MINIMUM permissions needed. A Lambda that reads from S3 should NOT have permission to delete databases. Start with zero access, add only what's needed.

🛡️ Security Layers

Network: VPC, Security Groups, NACLs
Identity: IAM, MFA, SSO
Data: Encryption at rest + in transit
Monitoring: CloudTrail, GuardDuty

Auto Scaling — Grow & Shrink Automatically 📈
More users? Add servers. Less users? Remove them. Pay only for what you need.
Watch the servers scale with traffic:
📈 Horizontal Scaling (Scale OUT)

Add MORE servers when busy, remove when quiet. Like opening more checkout lanes at a supermarket. The cloud default.

📊 Vertical Scaling (Scale UP)

Make one server BIGGER (more CPU/RAM). Like replacing a small truck with a bigger truck. Has limits — eventually you hit the max size.

💡 Scale triggers: CPU > 70%? Add 2 servers. Requests/sec > 10,000? Add 5. Queue depth > 1,000? Add workers. It's all rules-based or predictive (ML-powered).

Infrastructure as Code (IaC) 📝
Never click buttons in a console — define your infrastructure in CODE
❌ The Old Way (Click Ops)

Log into console → click "Create Server" → pick options → click "Create Database" → configure networking manually… 😩

Problems: not reproducible, no version history, easy to make mistakes, can't review changes

✅ IaC Way (Code It)

Write code describing what you want → run it → infrastructure appears. Same code → same result every time. Store in Git, review in PR!

Benefits: reproducible, version-controlled, peer-reviewed, automated

Terraform Example — Create an AWS S3 bucket + Lambda:
resource "aws_s3_bucket" "my_data" {
  bucket = "my-app-data-bucket"
  versioning { enabled = true }
  encryption { sse_algorithm = "AES256" }
}

resource "aws_lambda_function" "processor" {
  function_name = "data-processor"
  runtime       = "python3.11"
  handler       = "main.handler"
  memory_size   = 256
  timeout       = 30
}

CI (Continuous Integration)

Every code push triggers: build → run unit tests → lint → security scan. Fail fast, fix early. Developers merge to main multiple times per day.

CD (Continuous Deployment)

If CI passes → auto-deploy to staging → run integration tests → deploy to production. No human clicks. Code goes live in minutes, not weeks.

Deployment Strategies:
🔵🟢 Blue/Green: Run 2 identical envs. Switch traffic from blue to green instantly. Rollback = switch back.
🐤 Canary: Deploy to 5% of users first. Monitor errors. If OK → roll out to 100%. If bad → rollback.
🔄 Rolling: Replace servers one at a time. Always some servers serving traffic. Gradual transition.
Monitoring & Observability 📊
You can't fix what you can't see — the 3 pillars of observability
📈
Metrics

Numbers over time: CPU%, memory, request count, error rate, latency P99

CloudWatch | Cloud Monitoring | Prometheus + Grafana
📋
Logs

Text records of what happened: errors, requests, debug info. Searchable and filterable.

CloudWatch Logs | Cloud Logging | ELK / Loki
🔍
Traces

Follow one request across 10+ services: API → Auth → DB → Cache → Response. Find the bottleneck.

X-Ray | Cloud Trace | Jaeger / Tempo
🚨 Alerting — Don't stare at dashboards all day

Set rules: "If error rate > 5% for 3 minutes → page the on-call engineer via Slack/PagerDuty." The system watches itself. You sleep. Alerts wake you only when needed.

✅ The 4 Golden Signals (from Google SRE): Latency, Traffic, Errors, Saturation. Monitor these 4 and you cover 90% of issues.

💡 Top Cost-Saving Strategies

🎟️ Reserved Instances: Commit to 1-3 years → save 40-75%

🎰 Spot Instances: Use spare capacity → save up to 90% (can be interrupted!)

📏 Right-sizing: Most servers are over-provisioned. Downsize by 30-50%

⏰ Scheduling: Dev environments OFF at nights/weekends → save 65%

🗂️ Storage tiering: Move old data to Glacier (1/10th the price)

⚠️ Horror story: A startup left GPU instances running over a weekend = $72,000 bill. ALWAYS set billing alerts! (e.g., "Email me if bill > $1,000")
