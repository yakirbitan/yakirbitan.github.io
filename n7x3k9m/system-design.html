Step 0: The 4-Step Framework 🎯
Use this EXACT structure in EVERY system design interview — interviewers expect it
🧠 The #1 secret: System design is NOT about memorizing architectures. It's about structured thinking under ambiguity. The interviewer is evaluating HOW you think, not if you know the "right" answer. Drive the conversation. Ask questions. Make tradeoffs explicit.
1
Requirements & Scope (5 min)
Functional: What exactly does the system do? (e.g., "users can post tweets and follow other users")
Non-functional: Scale? Latency? Availability? Consistency? (e.g., "100M DAU, < 200ms reads, 99.99% uptime")
Out of scope: What are we NOT building? (e.g., "no DMs, no analytics for now")
2
Capacity Estimation (5 min)
Back-of-envelope math: QPS (queries/sec), storage, bandwidth.
Reads vs Writes? Read-heavy (100:1) → cache aggressively. Write-heavy (1:1) → focus on write throughput.
Storage: 100M users × 1KB per record = 100GB. × 5 years = 500GB → fits on one machine? Need sharding?
3
High-Level Design (15-20 min)
Draw the architecture: clients → load balancer → services → databases.
Define APIs. Pick database types. Identify core components. Keep it simple first.
4
Deep Dive + Tradeoffs (15-20 min)
The interviewer will pick 1-2 areas to go deep. Discuss tradeoffs, bottlenecks, how to scale, failure modes.
This is where you shine. Show depth of knowledge.

Building Block 1: API Design 🔌
Every system starts with its API — how clients talk to it
💡 Key idea: Before drawing boxes, define the APIs. This forces you to understand the requirements concretely. REST for CRUD operations, WebSockets for real-time, GraphQL for flexible queries. Always define: endpoint, method, request params, response, and rate limits.
REST API Example (Twitter)
# Create tweet POST /api/v1/tweets {"content": "Hello world", "media_ids": []} → 201 {"tweet_id": "abc123"} # Get timeline (paginated) GET /api/v1/timeline?cursor=xxx&limit=20 → 200 {"tweets": [...], "next_cursor": "yyy"} # Follow user POST /api/v1/users/{user_id}/follow → 200
Protocol Choices

REST — Standard CRUD. Stateless. Used by 90% of APIs.

WebSocket — Real-time bidirectional. Chat, notifications, live feeds.

gRPC — Service-to-service. Fast binary protocol. Used internally at Google.

GraphQL — Client specifies exactly what data it wants. Good for frontend flexibility.

SSE — Server-sent events. One-way server→client. Simpler than WebSocket for push notifications.

🔑 Always mention: Pagination (cursor-based > offset), rate limiting, authentication (JWT/OAuth), versioning (/v1/), idempotency keys for writes.
Building Block 2: Load Balancers ⚖️
Distribute traffic across multiple servers — the gateway to horizontal scaling
💡 Key idea: A load balancer sits between clients and servers. It distributes incoming requests so no single server gets overwhelmed. It also provides health checks — if a server dies, traffic routes to healthy ones automatically. This is HOW you scale: add more servers behind the LB.

Algorithms

Round Robin: 1→2→3→1→2→3. Simple, equal distribution.

Weighted Round Robin: More powerful server gets more requests.

Least Connections: Send to server with fewest active connections.

Layer 4 vs Layer 7

L4 (Transport): Routes based on IP/port. Fast but dumb. (AWS NLB)

L7 (Application): Routes based on URL, headers, cookies. Smarter, can do path-based routing. (AWS ALB, Nginx)

In interviews: say "Layer 7 load balancer" for most web apps.

Real-world: AWS ALB/NLB, Google Cloud LB, Nginx, HAProxy. Also mention DNS-level load balancing for global distribution.
Building Block 3: Databases 🗄️
SQL vs NoSQL — the most important choice in system design
💡 Key idea: SQL (relational) = structured data with relationships, ACID guarantees, complex queries. NoSQL = flexible schema, horizontal scaling, high throughput. Default to SQL unless you have a specific reason for NoSQL. The reason is always: massive scale, flexible schema, or specific access patterns.

NoSQL Types (know all 4!)

Key-Value: Redis, DynamoDB. Simple get/set by key. Fastest. Caching, sessions.

Document: MongoDB, Couchbase. JSON-like docs. Flexible schema. Product catalogs.

Wide-Column: Cassandra, HBase. Rows with dynamic columns. Time-series, IoT.

Graph: Neo4j, Neptune. Nodes + edges. Social networks, recommendations.

Interview Decision Framework

Need relationships/joins? → SQL

Need ACID transactions? → SQL

Massive writes (100K+/sec)? → NoSQL (Cassandra)

Flexible/evolving schema? → NoSQL (MongoDB)

Simple key-value + ultra fast? → Redis/DynamoDB

Graph-like queries (friends-of-friends)? → Graph DB

Building Block 4: Caching 🏎️
Store frequently accessed data in fast memory — the single biggest performance win
💡 Key idea: Cache = fast layer (RAM) in front of a slow layer (DB). If data is in cache (cache hit) → return instantly (~1ms). If not (cache miss) → query DB (~10-100ms), store result in cache for next time. A 90% hit rate means 90% of your reads skip the DB entirely.

Caching Strategies

Cache-Aside (Lazy): App checks cache → miss → reads DB → writes to cache. Most common.

Write-Through: Every write goes to cache AND DB simultaneously. Always consistent but slower writes.

Write-Behind: Write to cache first, async write to DB. Fast writes but risk of data loss.

Read-Through: Cache itself fetches from DB on miss. Simpler app code.

Cache Eviction

LRU (Least Recently Used) — evict oldest-accessed. Default choice.

LFU (Least Frequently Used) — evict least-used. Better for hot/cold data.

TTL (Time To Live) — expire after N seconds. Always set TTL as safety net.

⚠️ Cache Invalidation is the hard problem. When DB updates, how do you update/delete the cache? This is "one of the two hard things in CS."

Where to cache: Client (browser cache, CDN) → API Gateway → App-level (Redis/Memcached) → DB query cache. Multiple layers.
Building Block 5: CDN & Storage 🌐
Serve static content from the edge — close to the user
💡 Key idea: A CDN (Content Delivery Network) is a globally distributed cache for static files (images, videos, CSS, JS). Instead of every user fetching from your server in us-east-1, the CDN caches content at 200+ edge locations worldwide. User in Tokyo gets served from Tokyo PoP (~20ms) instead of Virginia (~200ms).
Push vs Pull CDN

Pull CDN: CDN fetches from origin on first request, then caches. Best for dynamic/frequently changing content. (CloudFront, Fastly)

Push CDN: You upload content to CDN proactively. Best for large static files you control (videos, software downloads).

Object Storage (S3)

For blobs: images, videos, files, backups. NOT for structured data.

Virtually unlimited storage. $0.023/GB/month.

Pattern: User uploads → store in S3 → serve via CDN.

DB stores metadata (filename, size, S3 URL), NOT the file itself.

Building Block 6: Message Queues 📬
Decouple services, handle bursts, enable async processing
💡 Key idea: A message queue sits between Producer and Consumer. The producer writes messages, the consumer processes them at its own pace. This gives you: (1) Decoupling — services don't need to know about each other, (2) Buffering — handle traffic spikes, (3) Async — don't block the user for slow work (send email, resize image, update search index).

Kafka vs Simple Queues

SQS/RabbitMQ: Simple job queue. Message consumed once and deleted. Task workers.

Kafka: Distributed log. Messages retained. Multiple consumer groups can read same data. Event streaming, real-time pipelines.

Rule of thumb: Task queue (resize image) → SQS. Event stream (user clicked X, propagate to 5 services) → Kafka.

Building Block 7: Database Scaling 📈
Replication, sharding, and partitioning — how to handle billions of rows
💡 Key idea: When one DB isn't enough: (1) Read Replicas — copies that handle read queries (scale reads). (2) Sharding — split data across multiple DBs by a shard key (scale writes + storage). Sharding is painful and the last resort. Always try: index optimization → read replicas → caching → vertical scaling → THEN sharding.

Shard key choice is CRITICAL. Bad key → uneven distribution (hotspots). Good keys: user_id, geographic region. Bad keys: timestamp (all writes go to latest shard).

Consistent Hashing (mention in interviews!)

When sharding, use consistent hashing so that adding/removing a shard only moves ~1/N of the data (instead of reshuffling everything). Nodes are placed on a hash ring. Each key maps to the next node clockwise. Adding a node only affects keys between it and its predecessor.

Core Concept: CAP Theorem & Consistency ⚖️
The fundamental tradeoff in distributed systems
💡 Key idea: In a distributed system, when a network partition happens, you MUST choose between Consistency (every read gets the latest write) and Availability (every request gets a response). You can't have both during a partition. CP = banking (prefer correct data). AP = social media (prefer being available).
CAP Breakdown

C — Consistency: All nodes see the same data at the same time.

A — Availability: Every request gets a response (even if not latest data).

P — Partition Tolerance: System works despite network splits. (Always true in real systems!)

So the real choice is: CP or AP.

Consistency Spectrum

Strong: Reads always return the latest write. Slow but correct. (CP: Zookeeper, etcd)

Eventual: Reads may return stale data, but will converge. Fast. (AP: Cassandra, DynamoDB)

Causal: Preserves cause→effect ordering. Middle ground. (MongoDB)

In interviews, say: "For this system, I'll choose [strong/eventual] consistency because..."

ACID vs BASE
ACID (SQL): Atomicity, Consistency, Isolation, Durability → strong guarantees, lower throughput. BASE (NoSQL): Basically Available, Soft state, Eventually consistent → high availability, higher throughput.
Building Block 8: Rate Limiting & Security 🛡️
Protect your system from abuse, DDoS, and misbehaving clients
💡 Key idea: Rate limiting caps how many requests a client can make in a time window. Without it, a single client can overwhelm your system. Implement at the API Gateway level. Common: 100 requests/minute per user. Return HTTP 429 (Too Many Requests) when exceeded.
Algorithms

Token Bucket: Tokens added at fixed rate. Each request costs 1 token. No tokens → rejected. Most popular. Allows short bursts.

Sliding Window Log: Track timestamps of requests. Count in rolling window. More precise but more memory.

Fixed Window Counter: Count per minute. Simplest. Edge case: bursts at window boundary.

Leaky Bucket: Requests enter bucket, processed at fixed rate. Smooths burst traffic.

Implementation

Where: API Gateway (Kong, Envoy) or middleware layer.

Storage: Redis (fast, distributed, INCR + EXPIRE).

Key: Rate limit by user_id, IP, API key, or combination.

Headers: Return X-RateLimit-Remaining, X-RateLimit-Reset.

Distributed rate limiting is harder — use Redis Lua scripts for atomic check-and-increment.

Classic Design 1: URL Shortener (TinyURL) 🔗
THE most common system design interview question — tests fundamentals
💡 Key idea: Map a long URL to a short code (7 chars, base62 = 62⁷ = 3.5 trillion URLs). Core challenge: generate unique short codes at scale. Two approaches: hash the URL (collision risk) or use a distributed ID generator (no collisions).

ID Generation Strategies

1. Hash + truncate: MD5/SHA256(url) → take first 7 chars (base62). Simple but collision possible → check DB, regenerate.

2. Counter-based: Auto-increment ID → base62 encode. No collision but predictable. Use distributed counter (Snowflake/UUID).

3. Pre-generated pool: Generate IDs in batch, store in Redis. Workers grab next available. No collisions, fast.

Read Flow (redirect)

1. User hits tinyurl.com/abc123

2. Check Redis cache → hit? Return 301.

3. Miss → query DB → store in cache.

4. Return 301 (permanent) or 302 (temporary) redirect.

301 = browsers cache the redirect (less server load).
302 = server sees every request (better for analytics).

Deep dive topics: Analytics (count clicks per URL), TTL/expiration, custom aliases, rate limiting creation.
Classic Design 2: Twitter / News Feed 🐦
The most important tradeoff: fan-out on write vs fan-out on read
💡 Key idea — THE tradeoff: When user A posts a tweet and A has 1M followers: Fan-out on Write = immediately push tweet to all 1M followers' timelines (fast reads, slow writes). Fan-out on Read = when each follower opens their feed, pull A's latest tweets (slow reads, fast writes). Hybrid (what Twitter does): fan-out on write for normal users, fan-out on read for celebrities.
Fan-out on Write (Push model)
User posts tweet
Get all followers from Follow table
Write tweet_id to each follower's timeline cache (Redis list)
Each follower's GET /timeline → read from pre-computed cache

✅ Reads are FAST (just read from cache) | ❌ Writes are expensive for celebrities (push to millions)

Fan-out on Read (Pull model)
User opens timeline
Get list of people I follow
Fetch latest tweets from each + merge + sort by time
Return top N tweets

✅ Writes are instant | ❌ Reads are slow (N queries + merge at read time)

Data Model
Users table: user_id, name, bio, ... (SQL)  |  Tweets table: tweet_id, user_id, content, timestamp, media_urls (SQL/NoSQL)  |  Follow table: follower_id, followee_id (SQL)  |  Timeline cache: user_id → [tweet_id, tweet_id, ...] (Redis sorted set by timestamp)

Classic Design 4: YouTube / Video Streaming 📹
Upload, transcode, store, and stream video at scale
💡 Key idea: Video is HUGE (a 1-hour 1080p video ≈ 1-3 GB). You can't serve raw uploads. The pipeline: (1) Upload to object storage, (2) Transcode to multiple resolutions (240p, 480p, 720p, 1080p) + formats (H.264, VP9), (3) Store transcoded chunks on object storage, (4) Serve via CDN using adaptive bitrate streaming (HLS/DASH) — client progressively downloads chunks at the best quality for their bandwidth.
Upload & Processing Pipeline
Upload: Client → pre-signed S3 URL → direct upload to object storage (bypasses app server)
Queue: Upload event → message queue (Kafka/SQS)
Transcode: Workers split video into chunks → encode each in multiple resolutions → parallel processing
Store: Chunks stored in S3 → CDN URL generated → DB updated with metadata
Serve: Client requests manifest (HLS .m3u8) → CDN serves video chunks adaptively
Adaptive Bitrate (HLS/DASH)
Video split into 2-10 second chunks. Client measures bandwidth and switches quality seamlessly. Buffering? Switch to 480p. Good WiFi? Switch to 1080p. Each chunk is a separate CDN-cached file.
Storage Estimation
500 hours uploaded/minute (YouTube)
× 3 resolutions × 0.5 GB avg = 750 GB/minute
× 1440 min/day = ~1 PB/day
This is why CDN + object storage is critical.

Classic Design 6: Search & Autocomplete 🔍
Full-text search with Elasticsearch + typeahead suggestions
💡 Key idea — Search: Use an inverted index (Elasticsearch/Solr). Instead of "doc → words", store "word → [doc1, doc3, doc7]". Query = find intersection of posting lists. Relevance = TF-IDF or BM25 scoring.
💡 Key idea — Typeahead: Use a Trie in memory that stores top-K completions at every prefix node. User types "app" → traverse trie to "app" node → return precomputed ["apple", "application", "app store"]. Update trie periodically from analytics.
Search Architecture (Elasticsearch)
Data changes in primary DB (MySQL)
CDC (Change Data Capture) or queue event → update Elasticsearch index
Search query → Elasticsearch (inverted index, < 50ms for millions of docs)
ES returns ranked doc IDs → app fetches full data from DB/cache

Key: ES is NOT the source of truth. Primary DB is. ES is a read-optimized index.

Typeahead Design

Data structure: Trie where each node stores top-K completions (precomputed).

Serving: Trie fits in memory (100M queries × 20 chars = ~2GB). Replicate across multiple servers.

Updates: Collect search analytics → offline pipeline (MapReduce) → rebuild trie weekly → swap in.

Latency: < 100ms requirement. Debounce on client side (only query after 300ms pause).

Classic Design 7: Distributed Key-Value Store 🔑
The building block behind DynamoDB, Cassandra, and distributed caches
💡 Key idea: This is where all the distributed systems theory comes together. A distributed KV store must handle: (1) Partitioning — which node stores which key? (consistent hashing), (2) Replication — copy to N nodes for durability, (3) Consistency — quorum reads/writes (W+R > N guarantees consistency), (4) Failure detection — gossip protocol, (5) Conflict resolution — vector clocks or last-write-wins.
Core Mechanisms

Consistent Hashing: Hash ring. Each node owns a range. Adding/removing node only affects neighbors.

Replication: N=3 → store each key on 3 consecutive nodes clockwise on the ring.

Quorum: W=2, R=2, N=3. Write succeeds when 2/3 nodes ACK. Read queries 2/3 nodes. W+R>N → guaranteed to see latest write.

Gossip Protocol: Nodes periodically share state with random peers. Failure detected when no heartbeat.

Tunable Consistency

Strong (W=N, R=1): All nodes must ACK write. Slow but always consistent.

Eventual (W=1, R=1): Write to any one node, read any one. Fastest but may be stale.

Balanced (W=2, R=2, N=3): Good balance. Default recommendation.

Conflict resolution: Vector clocks track causality. If no causal relation → conflict → application must resolve (or use last-write-wins / CRDTs).

Classic Design 8: Web Crawler 🕷️
Crawl billions of pages politely and efficiently
💡 Key idea: A web crawler is a BFS on the internet graph. Start with seed URLs → fetch page → extract links → add new links to queue → repeat. Core challenges: (1) Politeness — don't overwhelm any single host (rate limit per domain), (2) Deduplication — don't crawl the same URL twice (Bloom filter), (3) Scale — billions of URLs across thousands of workers, (4) Freshness — re-crawl pages that change frequently.
Architecture
URL Frontier (priority queue): URLs to crawl, prioritized by importance + freshness
DNS Resolver: Resolve domain → IP (cache heavily)
Fetcher Workers: Download HTML. Obey robots.txt. Rate limit per domain (1 req/sec/domain).
Content Parser: Extract links, text. Detect duplicates via content hash (SimHash for near-duplicates).
URL Dedup: Bloom filter (probabilistic, O(1) lookup, 1% false positive OK). Seen before? Skip.
Storage: Store page content in object storage, metadata in DB.
Bloom Filter: A space-efficient probabilistic data structure. Can tell you "definitely not in set" or "probably in set." Perfect for URL dedup — 10 billion URLs in ~1.2 GB of memory.
Classic Design 9: Uber / Google Maps (Proximity) 📍
Location-based services — find nearby drivers, restaurants, friends
💡 Key idea: Searching "all drivers within 5km of me" on a flat DB is O(n) — too slow. Use a spatial index. Two approaches: (1) Geohash — divide earth into grid cells, encode as string. Nearby locations share prefix. Store in Redis as sorted set. (2) QuadTree — recursively subdivide areas into 4 quadrants until each cell has < K points. Both give O(log n) spatial queries.
Geohash (preferred in interviews)

Encode (lat, lng) into a string like "9q8yyk"

Longer string = more precise grid cell

6 chars ≈ ±1.2 km (good for nearby drivers)

Nearby points share prefix → efficient DB prefix query

Store: Redis GEOADD drivers lng lat driver_id

Query: Redis GEORADIUS drivers lng lat 5 km

Uber Ride Matching Flow
Rider requests ride (lat, lng, destination)
Query spatial index: find available drivers within 5km
Ranking service: sort by distance, rating, ETA
Send ride request to best driver via WebSocket
Driver accepts → match confirmed → start tracking

Location updates: Drivers send GPS every 3-5 sec → update spatial index. High write throughput → Redis or in-memory QuadTree.

Classic Design 10: Google Drive / Dropbox 📁
File sync, deduplication, and conflict resolution
💡 Key idea: The hard part isn't storage (use S3). It's sync. When user edits a file on laptop, it must appear on phone. Approach: split files into chunks (4MB each). Only upload changed chunks (dedup!). Use a metadata DB to track file → chunks mapping. Long-polling or WebSocket to notify other devices of changes. Conflict? Keep both versions and let user resolve.
Upload Flow (chunked + dedup)
Client splits file into 4MB chunks
Hash each chunk (SHA256)
Ask server: "which chunks do you already have?"
Upload only NEW chunks to block storage (S3)
Update metadata: file → [chunk_hash_1, chunk_hash_2, ...]

Huge win: Edit 1 line in a 100MB file → only re-upload 1 chunk (4MB), not 100MB.

Sync & Conflict Resolution

Notification: When file changes, server notifies all user's devices via long-polling / WebSocket.

Versioning: Every edit creates a new version. Old versions kept for 30 days.

Conflict: Two devices edit same file offline. On sync, server detects divergence → keeps both → user resolves.

Operational Transform / CRDT: For real-time collaborative editing (Google Docs), more advanced.

Design Patterns: Microservices Essentials ⚙️
Patterns that come up in every deep-dive: service discovery, circuit breaker, idempotency
💡 Key idea: In a microservices world, services fail independently. You need patterns for resilience: Circuit Breaker (stop calling a failing service), Retry with backoff (handle transient failures), Idempotency (safe to retry without double-processing), Saga (distributed transactions across services).
Circuit Breaker

States: Closed (normal) → Open (after N failures, stop calling) → Half-Open (try one request, if succeeds → close)

Prevents cascade failures. If Payment Service is down, don't keep hammering it — fail fast and return cached/default response.

Saga Pattern (Distributed Tx)

Order Service → Payment Service → Inventory Service. If payment fails, need to undo order.

Choreography: Each service emits events. Others react. Decoupled but hard to track.

Orchestration: A central "saga coordinator" tells each service what to do/undo. Easier to reason about.

Idempotency
Client sends Idempotency-Key: abc123 header. Server stores result in Redis keyed by idempotency key. If same key arrives again → return cached result, don't process twice. Critical for payments (never charge twice!).
Service Discovery
How does Service A find Service B's IP? Service Registry (Consul, etcd, Zookeeper). Services register on startup, deregister on shutdown. Or use DNS-based (Kubernetes built-in: service-b.namespace.svc.cluster.local).

🎯 The Ultimate Rule

System design is about structured communication, not memorization. Always: (1) Clarify requirements, (2) Do back-of-envelope math, (3) Start simple → evolve, (4) Discuss tradeoffs explicitly. The interviewer wants to see HOW you think, not WHAT you memorize.
