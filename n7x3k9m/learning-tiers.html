T6

✅ Easy — Start Here

You likely know 80% of this already. Blast through in 1–2 days. If anything takes >2h → it was misclassified for you.

⏱ 1–2 days total
#1
DS&A
Two Pointers
Use two indices moving toward each other or in the same direction to reduce O(n²) brute force to O(n).
Key insight: Works on sorted arrays/strings. One pointer from each end, or both from the left at different speeds. Never nested loops if you can use this.
Examples: Two Sum (sorted), Valid Palindrome, Container With Most Water, 3Sum, Remove Duplicates
#2
DS&A
Prefix Sum
Precompute cumulative sums so any range query [i,j] answers in O(1) instead of O(n).
Key insight: rangeSum(i,j) = prefix[j+1] − prefix[i]. Build prefix array once, query forever. Also works for 2D grids.
Examples: Subarray Sum Equals K, Range Sum Query, Product Array, Running Sum
#3
DS&A
HashMap / HashSet
Trade O(n) space for O(1) average lookup, insert, and existence checks. The universal "seen before?" tool.
Key insight: Anytime you catch yourself writing a nested loop to check "have I seen this?", you need a HashMap. Complements Two Pointers and Sliding Window.
Examples: Two Sum, Group Anagrams, Top K Frequent, LRU Cache, Longest Consecutive Sequence
#4
DS&A
Bit Manipulation
Use bitwise operations (XOR, AND, OR, shifts) to solve problems with O(1) space and no extra data structures.
Key insight: XOR of a number with itself = 0. XOR with 0 = itself. a & (a-1) clears the lowest set bit. n & 1 checks odd/even. Fixed set of ~10 tricks covers everything.
Examples: Single Number, Missing Number, Number of 1 Bits, Power of Two, Reverse Bits
#5
System Design
URL Shortener
Map long URLs to short codes. Two approaches: hash (MD5/SHA-256 → take first 7 chars, handle collisions) or counter + base62 encode.
Key insight: Counter + base62 is simpler and collision-free. 301 (permanent) vs 302 (temporary) redirect matters for SEO and caching. Cache hot URLs in Redis with TTL.
System deep-dives: 301 vs 302 tradeoff · Distributed counter (Snowflake ID) · Read-heavy → cache aggressively
#6
System Design
API Design
Choose and design the right communication protocol: REST, gRPC, WebSocket, or GraphQL based on the use case.
Key insight: REST = stateless CRUD. gRPC = low-latency internal microservices (binary, streaming). WebSocket = bidirectional real-time (chat, gaming). GraphQL = flexible queries (mobile clients).
Apply to: All SD problems — always define API before drawing boxes
#7
System Design
CDN & Load Balancing
CDN caches static content at edge nodes close to users. Load balancers distribute traffic across server pools.
Key insight: Push CDN (you control what's cached) vs Pull CDN (cache on first request). L4 LB = TCP level, fast. L7 LB = HTTP level, smart routing. Algorithms: round-robin, least connections, IP hash (sticky sessions).
Apply to: YouTube, Twitter, any read-heavy system with global users
#8
System Design
Caching Strategies
Store frequently accessed data in memory (Redis/Memcached) to reduce latency and DB load.
Key insight: Cache-aside (app loads cache on miss) = most common. Write-through (write DB+cache together) = strong consistency. LRU eviction = evict least recently used. Always ask: what's the invalidation strategy?
Apply to: Every read-heavy system. Twitter timeline, URL Shortener, session storage, rate limiting counters
T5

🟢 Medium-Easy — Warm Up

Patterns you know conceptually but need to drill. Some variants will surprise you. Spend 2–4 days here.

⏱ 2–4 days total
#9
DS&A
Sliding Window (Fixed)
Maintain a window of fixed size k, sliding right one element at a time. Update window state by adding right, removing left.
Key insight: What you track in the window (sum, max, character counts) is everything. The window itself is trivial — the state management inside is the challenge.
Examples: Max Sum Subarray of Size K, Find All Anagrams, Sliding Window Maximum
#10
DS&A
Binary Search (Standard + Rotated)
O(log n) search in sorted array. Standard, rotated array, find peak element, find minimum in rotated.
Key insight: Always define your invariant: what does mid being "too small" or "too large" really mean? For rotated: determine which half is sorted, then decide which half to search.
Examples: Binary Search, Search in Rotated Array, Find Peak Element, Search a 2D Matrix, Koko Eating Bananas
#11
DS&A
Trees — BFS (Level Order)
Process a tree level by level using a queue. Capture all nodes at depth d before moving to depth d+1.
Key insight: Snapshot queue.length at the start of each level — that tells you how many nodes belong to this level. Enables zigzag traversal, right-side view, max width, etc.
Examples: Level Order Traversal, Zigzag Traversal, Right Side View, Max Width, Connect Level-Order Nodes
#12
DS&A
Linked List (Slow/Fast Pointer)
Floyd's algorithm: two pointers at different speeds. Fast (2x) catches up to slow, revealing cycles, midpoints, Nth-from-end.
Key insight: If fast and slow meet → cycle exists. When fast hits null → slow is at midpoint. To find nth from end: fast ahead by n steps, then move both at same speed.
Examples: Linked List Cycle, Find Middle, Remove Nth Node From End, Palindrome Linked List, Reorder List
#13
DS&A
Intervals
Problems involving overlapping ranges. Sort by start time, then merge, insert, or count overlaps.
Key insight: Overlap condition: A.end ≥ B.start (when sorted). Merge: new interval = [min(starts), max(ends)]. For "minimum rooms needed" → sort starts and ends separately, simulate.
Examples: Merge Intervals, Insert Interval, Non-Overlapping Intervals, Meeting Rooms II, Employee Free Time
#14
System Design
Rate Limiting
Control how many requests a client can make in a time window. Protect services from abuse and overload.
Key insight: Token Bucket (smooth bursts, most common). Sliding Window Log (precise, memory heavy). Fixed Window (simple, edge cases at window boundary). Redis INCR + EXPIRE = simplest distributed implementation.
System deep-dives: Where to enforce (gateway vs service) · Distributed rate limiting across nodes · Client fingerprinting (IP, user ID, API key)
#15
System Design
Database Replication
Copy data to multiple nodes to improve read performance and availability. Leader-follower (primary-replica) is the standard model.
Key insight: Sync replication = strong consistency, higher latency. Async replication = faster writes, risk of data loss on failover. Read from replicas = eventual consistency (replication lag). Multi-leader = conflict resolution needed.
Apply to: Any high-read system. PostgreSQL streaming replication, MySQL binlog, MongoDB replica sets
T4

🔵 Medium — The Real Foundation

This tier takes the longest relative to "assumed knowledge". Patterns that look familiar but have non-obvious variants. 3–5 days.

⏱ 3–5 days total
#16
DS&A
Sliding Window (Variable)
Window size changes dynamically: expand right when constraint is satisfied, shrink from left when violated. Maximizes/minimizes window under a constraint.
Key insight: Template: expand right unconditionally. When constraint is violated → shrink left until valid. Track state with a hashmap or counter. The "when to shrink" condition is the only thing that changes per problem.
Examples: Longest Substring Without Repeating, Min Window Substring, Longest Subarray Sum ≤ K, Fruit Into Baskets
#17
DS&A
Binary Search on Answer
Don't search in the input array — search in the answer space. Binary search on what the answer could be, then verify with a helper function.
Key insight: If the problem asks "what's the minimum X such that Y is possible?" → binary search on X. Helper function: "can we achieve X?" returns true/false. This transforms many hard problems into medium.
Examples: Koko Eating Bananas, Capacity to Ship Packages, Split Array Largest Sum, Find Minimum in Rotated (answer variant)
#18
DS&A
Graph — BFS (Shortest Path)
BFS finds the shortest path in unweighted graphs. Multi-source BFS starts from multiple nodes simultaneously. 0-1 BFS uses a deque for edge weights of 0 or 1.
Key insight: Always use a visited set — without it BFS loops infinitely. Multi-source: add all sources to queue at the start (level 0). 0-1 BFS: push front for 0-weight edges, push back for 1-weight edges.
Examples: Word Ladder, Rotting Oranges, Shortest Path in Binary Matrix, 0-1 Matrix, Walls and Gates
#19
DS&A
Trees — DFS (Core)
Recursive traversal passing information down (parameters) and collecting information up (return values) through the tree.
Key insight: Down = constraints/state from root. Up = aggregated result from subtrees. For "max path sum through any node": each node returns best single path, but tracks global max using left + current + right locally.
Examples: Max Depth, Same Tree, Invert Tree, Path Sum II, Max Path Sum, Validate BST, Diameter of Binary Tree
#20
DS&A
Greedy
At each step, pick the locally optimal choice. Works when local optimality leads to global optimality — but you must prove it (or at least convince yourself).
Key insight: The hard part is knowing WHEN greedy is valid. Use exchange argument: "if I swap my greedy choice with any other choice, can it ever be better?" If no → greedy is correct. Common patterns: sort by one dimension, pick max/min greedily.
Examples: Jump Game, Gas Station, Assign Cookies, Task Scheduler, Non-Overlapping Intervals, Partition Labels
#21
System Design
YouTube / Video Streaming
Large file upload → transcoding pipeline → CDN distribution. The key challenge is the video processing pipeline and adaptive streaming.
Key insight: Upload to S3 → trigger transcoding workers (encode to multiple resolutions/formats) → store in S3 → serve via CDN. HLS/DASH = adaptive bitrate (client switches quality based on bandwidth). Content stored geographically near viewers.
System deep-dives: Pre-signed S3 uploads · Transcoding with FFmpeg workers · Thumbnail generation · View count (Redis counter → async write)
#22
System Design
Cassandra Data Modeling
Design tables around query patterns, not relationships. In Cassandra, you denormalize by design — duplicate data to make reads fast.
Key insight: Partition key = which node holds the data (must distribute evenly, must be in your WHERE clause). Clustering key = sort order within partition. Design one table per query. Never do full table scans — always query by partition key.
Apply to: Chat history (user_id + timestamp), IoT sensor data, time-series, leaderboards, activity feeds
T3

🟡 Medium-Hard — Requires Real Effort

These won't click on first read. Need multiple passes + implementation. Plan for 4–7 days across this tier.

⏱ 4–7 days total
#23
DS&A
Trie (Prefix Tree)
Tree where each node represents one character. All words sharing a prefix share the same path from root. O(m) insert/search where m = word length.
Key insight: Each TrieNode has children[26] (or HashMap) + isEndOfWord flag. Variants: wildcard search (DFS on '.' nodes), XOR trie for max XOR (insert binary representation), compressed trie for space efficiency.
Examples: Implement Trie, Word Search II, Design Add Search Words, Replace Words, Maximum XOR of Two Numbers
#24
DS&A
Monotonic Stack
Stack that maintains a strictly increasing or decreasing order of elements. Pops elements that violate the ordering to find next greater/smaller.
Key insight: When you pop an element, the current element is its "next greater" (or smaller). Maintain indices, not values. For circular arrays: process 2n elements with index mod n. Hardest part: recognizing the problem needs this at all.
Examples: Next Greater Element, Daily Temperatures, Largest Rectangle in Histogram, Trapping Rain Water, Sum of Subarray Minimums
#25
DS&A
Trees DFS — Advanced (LCA, Tree DP)
Lowest Common Ancestor, tree diameter, binary lifting for fast LCA, tree DP (solve subproblems at each subtree). Requires composing results from children.
Key insight: LCA: a node is LCA if one target in left subtree and one in right (or node is itself a target). Tree DP: define what each subtree "returns" — often (maxDepth, hasAnswer). Binary lifting: precompute 2^k ancestors for O(log n) LCA queries.
Examples: Lowest Common Ancestor, Diameter of Binary Tree, Binary Tree Cameras, House Robber III, Kth Ancestor
#26
System Design
Chat System / WhatsApp
Real-time bidirectional messaging at scale. Three hard sub-problems: delivery protocol, message persistence, online presence.
Key insight: WebSocket = persistent TCP connection for push delivery (not polling). Presence via heartbeat (client pings every 5s, server marks offline after 30s miss). Message storage in Cassandra: partition by (sender, receiver) + cluster by timestamp. Fan-out to group members via message queue.
System deep-dives: Message deduplication (client-side ID) · E2E encryption · Read receipts · Offline message queue · Group chat fan-out
#27
System Design
Distributed Search / Typeahead
Two distinct problems: full-text search (inverted index) and autocomplete suggestions (Trie or top-k precomputed). Both need to work at massive scale.
Key insight: Full-text search: Elasticsearch uses inverted index (word → list of document IDs). Typeahead: Trie for prefix matching + top-k results per node (precomputed). Cache suggestions aggressively — same prefix hit millions of times per hour. Update trie async, not on every request.
System deep-dives: Ranking signals (freshness, clicks, personalization) · Fuzzy search (edit distance) · Sharding the index · Typeahead at Google scale
#28
System Design
Uber / Proximity Service
Find drivers near a rider. Core challenge: efficiently querying nearby entities on a 2D map, updated in real time.
Key insight: Geohash: encode lat/lng into a base32 string — nearby locations share common prefixes. QuadTree: recursive 2D partitioning, handles uneven density. For real-time: drivers push location every few seconds → update grid cell → query nearby cells. Ride matching via pub/sub (notify drivers in geo-cell).
System deep-dives: Geohash precision levels · Neighboring cell search · ETA calculation (road graph Dijkstra) · Surge pricing triggers
#29
System Design
Google Drive / File Sync
Reliable cloud file storage with sync across devices. Hard problems: chunking, dedup, delta sync, and conflict resolution.
Key insight: Chunk files (4MB blocks) + hash each chunk → detect unchanged chunks → only upload changed chunks (delta sync). Dedup: if chunk hash already exists in store, skip upload. Conflict resolution: vector clocks or "last write wins" per file version. Client maintains local metadata DB (SQLite).
System deep-dives: Block server vs metadata server separation · Offline edits sync · Notification service for updates · Versioning with S3 versioning

🔴 Hard — Fight Through These

Each of these genuinely requires 1–3 days of focused effort. Don't rush — the depth is the point. 1–2 weeks total.

⏱ 1–2 weeks total
#30
System Design
DynamoDB — Single-Table Design
Model all entities from your domain into ONE DynamoDB table. Access patterns drive key design — completely opposite to relational modeling.
Key insight: Define all access patterns FIRST. Then design PK/SK to answer each. Use prefixes to encode entity type (PK="USER#123", SK="ORDER#456"). GSI (Global Secondary Index) = new PK/SK pair to support additional query patterns. The hardest mindset shift: you cannot do ad-hoc queries — every query must be planned.
System deep-dives: Hot partition avoidance with write sharding · LSI vs GSI tradeoffs · DynamoDB Streams for CDC · Single-table patterns for 1:M, M:M relationships
#31
DS&A
Topological Sort
Linear ordering of nodes in a DAG such that for every directed edge u→v, u comes before v. Two implementations: Kahn's (BFS + in-degree) and DFS-based.
Key insight: Kahn's: count in-degrees, add 0-in-degree nodes to queue, process and decrement neighbors — if remaining count > 0 at end, there's a cycle. DFS-based: post-order DFS, add to front of result. Both do cycle detection. Real use: build systems, course prerequisites, task scheduling.
Examples: Course Schedule I & II, Alien Dictionary, Sequence Reconstruction, Minimum Height Trees, Find Order
#32
DS&A
Union-Find (Disjoint Set)
Tracks a set of elements partitioned into disjoint subsets. Supports two operations: find (which set?) and union (merge sets). With optimizations: nearly O(1) per operation.
Key insight: Path compression: when finding root, point all nodes directly to root. Union by rank: always attach smaller tree under larger. Together → amortized O(α(n)) ≈ O(1). Use when: "are these two nodes connected?" or "how many connected components?"
Examples: Number of Connected Components, Redundant Connection, Accounts Merge, Graph Valid Tree, Smallest String With Swaps
#33
System Design
Database Sharding
Horizontal partitioning: split data across multiple DB nodes by a shard key. Each shard holds a subset of the data.
Key insight: Shard key choice is everything — bad key = hot spots (all traffic goes to one shard). Good key = uniform distribution. Range-based sharding = easy range queries but uneven load. Hash-based = uniform load but range queries hit all shards. Consistent hashing = node add/remove only moves K/N keys (minimal reshuffling).
System deep-dives: Resharding strategies · Cross-shard joins (denormalize or application layer) · Celeb problem (hot user shards) · MongoDB sharding vs Vitess (MySQL)
#34
DS&A
Heap — Advanced Patterns
Two complementary heaps for median stream. k-way merge. Skyline problem. Scheduling. These go far beyond basic "top K elements".
Key insight: Two heaps for median: max-heap (left half) + min-heap (right half). Keep sizes balanced ±1. Top of max-heap and top of min-heap straddle the median. K-way merge: put first element of each list in a min-heap with list pointer — always pop min and advance that list. Skyline: event-based sweep + max-heap.
Examples: Find Median from Data Stream, Merge K Sorted Lists, Ugly Number II, Task Scheduler, The Skyline Problem
#35
System Design
Twitter / News Feed
Generate a ranked feed of posts from accounts a user follows. Scale challenge: celebrities with millions of followers, real-time freshness, ranking.
Key insight: Fan-out on write (push): when user posts, write to all followers' feed caches immediately. Fast reads, expensive writes, breaks for celebrities. Fan-out on read (pull): build feed on-demand by fetching from all followees. Expensive reads, cheap writes. Hybrid: push for regular users, pull for celebrity posts at read time. Merge in application layer.
System deep-dives: Redis sorted set for feed cache (score=timestamp) · Ranking signals (recency, engagement) · Feed pagination (cursor-based) · Real-time notifications via WebSocket
#36
System Design
Web Crawler
Systematically browse the web: start from seed URLs, fetch pages, extract links, add new URLs to frontier, repeat. Scale: billions of pages.
Key insight: BFS (breadth-first) for balanced coverage. Bloom filter to check "have I seen this URL?" without storing all URLs in memory. Politeness: per-domain rate limiting + respect robots.txt. URL normalization: lowercase, strip fragments, canonical form. Distributed crawler: partition frontier by domain hash to avoid politeness conflicts.
System deep-dives: Trap detection (infinite URL spaces) · Content dedup (SimHash) · Recrawl priority (freshness vs importance) · URL frontier as priority queue (PageRank score)

Tier 1 — Brutal. Don't rush these.

Deep distributed systems theory + the hardest algorithmic patterns. Context from Tiers 2–6 makes these stick. 2–3 weeks total.

⏱ 2–3 weeks total
#37
System Design
Consistent Hashing + Distributed KV Store
Distribute data across N nodes with minimal reshuffling when nodes join/leave. The foundation of how Cassandra, DynamoDB, and Redis Cluster work.
Key insight: Hash ring: map both nodes and keys to a circle. Each key belongs to the first node clockwise. Virtual nodes (each physical node = multiple ring positions) → even distribution. Quorum: R + W > N guarantees overlap (e.g. N=3, R=2, W=2). Gossip protocol: nodes share state with k random peers every T seconds → eventually consistent membership. Vector clocks: track causality for conflict detection (not just timestamps).
System deep-dives: Consistent hashing rehashing vs naive modulo · Sloppy quorum + hinted handoff · Merkle trees for anti-entropy sync · Cassandra vs Dynamo vs Redis Cluster comparison
#38
System Design
CAP Theorem + Consistency Models
In a distributed system facing network partition, you must choose: Consistency (all nodes see same data) or Availability (every request gets a response). Both is impossible.
Key insight: CP systems (Zookeeper, HBase): sacrifice availability during partition → return error instead of stale data. AP systems (Cassandra, DynamoDB): sacrifice consistency → return possibly stale data. PACELC extends CAP: even without partition, choose latency vs consistency. Consistency spectrum: Strong → Sequential → Causal → Read-your-writes → Eventual. Each step is a different guarantee, not just "strong or weak".
System deep-dives: Why "BASE" complements ACID · Cassandra's tunable consistency (ONE/QUORUM/ALL) · Read-your-writes in social media · Causal consistency in Google Spanner
#39
DS&A
Graph — Advanced Algorithms
Dijkstra (shortest path, non-negative weights), Bellman-Ford (negative weights + cycle detection), Tarjan (SCCs + bridges), Floyd-Warshall (all pairs).
Key insight: Dijkstra: greedy approach with min-heap — only correct with non-negative weights. Bellman-Ford: relax all edges V-1 times → handles negative weights, detects negative cycle on Vth pass. Tarjan: DFS + low-link values to find strongly connected components and bridges. Each algorithm solves a distinct problem — knowing which to reach for is half the battle.
Examples: Network Delay Time (Dijkstra), Cheapest Flights Within K Stops (Bellman-Ford), Critical Connections (Tarjan Bridges), Course Schedule SCC
#40
DS&A
Backtracking + Pruning
Explore the entire decision tree recursively, undoing choices when they lead to dead ends. Pruning cuts branches early to avoid exploring impossible states.
Key insight: Template: choose → explore → unchoose. Three things to define: State (what snapshot of the problem looks like), Choices (options available at each step), Constraint (when to prune). Pruning is everything — without it, backtracking = brute force. Common pruning: sort + skip duplicates, check constraint before recursing, track remaining budget.
Examples: Subsets I & II, Permutations, Combination Sum, N-Queens, Sudoku Solver, Palindrome Partitioning, Word Search
#41
System Design
LSM Tree vs B-Tree Internals
The two dominant storage engine data structures. Understanding them explains why certain databases are write-optimized (Cassandra, RocksDB) vs read-optimized (PostgreSQL, MySQL).
Key insight: B-Tree: balanced tree, in-place updates, O(log n) read/write, page-based storage. PostgreSQL uses this. Good for mixed read/write workloads. LSM (Log-Structured Merge): writes go to in-memory MemTable → flushed as immutable SSTables → background compaction merges. Write amplification is lower (sequential writes), read amplification is higher (check multiple levels). Cassandra, RocksDB, LevelDB all use LSM. Bloom filters on SSTables reduce read overhead.
System deep-dives: Write amplification vs read amplification tradeoff · WAL (Write-Ahead Log) in both · Compaction strategies (size-tiered, leveled) · Why MongoDB moved from MMAPv1 to WiredTiger (B-tree)
#42
System Design
Distributed Consensus (Raft / Paxos)
How a cluster of nodes agrees on a single value despite failures. The algorithm that makes distributed databases consistent and fault-tolerant.
Key insight: Raft: leader election (highest term wins), log replication (leader sends entries to majority), commit (applied after majority ack). Split-brain prevention: only ONE leader per term, and a node only votes once per term. Leader lease: leader holds authority for a duration, doesn't need constant re-election. Why this matters: etcd uses Raft, CockroachDB uses Raft per range, Kafka (KRaft mode) replaced ZooKeeper with Raft.
System deep-dives: Leader election timeout randomization · Log compaction (snapshots) · Raft vs Paxos complexity · ZooKeeper ZAB vs Raft · Why Paxos is hard to implement correctly
